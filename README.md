# Auth API

> Production-ready authentication backend · Hono · Bun · TypeScript · Drizzle · PostgreSQL

---

## What is this?

A self-contained REST API that handles the full authentication lifecycle for a Next.js frontend.
It issues **httpOnly JWT cookies** (no localStorage, no headers) and silently rotates tokens so
users stay logged in for up to a year without ever seeing a login prompt.

The frontend contract is **frozen** — every field name, casing, status code, and cookie flag
is locked in and consumed directly by a typed Next.js app.

## Tech Stack

| Concern    | Tool                  | Why                                                             |
| ---------- | --------------------- | --------------------------------------------------------------- |
| Runtime    | **Bun**               | Fast startup, built-in test runner, native argon2 bindings      |
| Framework  | **Hono**              | Edge-compatible, zero-overhead, typed middleware                |
| Language   | **TypeScript** strict | `noImplicitAny`, `strictNullChecks`, `noUncheckedIndexedAccess` |
| ORM        | **Drizzle**           | Type-safe SQL, zero runtime overhead, migration CLI             |
| Database   | **PostgreSQL**        | Sessions table for token revocation                             |
| Passwords  | **Argon2id**          | Industry-best hashing via `@node-rs/argon2` (Bun-native)        |
| JWT        | **jose**              | Web Crypto API — Bun-native, no Node crypto dep                 |
| Validation | **Zod**               | Every env var, every request body, every DB row                 |

## File Serving — Two Roots

The server exposes two completely separate static-file roots.

### `/static/*` → `public/assets/`

Binary assets: images, icons, fonts, downloadable files.

```
GET /static/images/logo.png   → public/assets/images/logo.png
GET /static/icons/favicon.svg → public/assets/icons/favicon.svg
GET /static/fonts/inter.woff2 → public/assets/fonts/inter.woff2
```

Cache-Control: `public, max-age=2592000, immutable` (30 days in production).
Use content-addressed filenames (e.g. `logo.abc123.png`) so you can set immutable safely.

### `/views/*` → `src/assets/`

HTML templates, co-located CSS and JavaScript.

```
GET /views/css/app.css     → src/assets/css/app.css
GET /views/js/htmx.min.js  → src/assets/js/htmx.min.js
```

Cache-Control: `public, max-age=300` (5 minutes in production).
No-cache in development.

### `GET /` — Homepage

The root route reads `src/assets/html/homepage.html` at startup and serves it via `c.html()`.

```typescript
const homepage = readFileSync(pathConfig.htmlPages.homepage, "utf-8");
app.get("/", (c) => c.html(homepage));
```

## API Reference

### Auth Endpoints

All auth routes are prefixed `/api/auth`.

| Method | Path                | Auth required    | Description                                                    |
| ------ | ------------------- | ---------------- | -------------------------------------------------------------- |
| `POST` | `/api/auth/signup`  | No               | Create account. Sets `access_token` + `refresh_token` cookies. |
| `POST` | `/api/auth/signin`  | No               | Login. Sets cookies.                                           |
| `POST` | `/api/auth/signout` | ✅               | Revoke session. Clears cookies.                                |
| `POST` | `/api/auth/refresh` | No (uses cookie) | Rotate tokens silently.                                        |
| `GET`  | `/api/auth/me`      | ✅               | Return current user.                                           |

### Response Envelope

Every response — success or error — uses the same flat shape:

**Success:**

```json
{
  "success": true,
  "message": "Login successful",
  "user": { ... },
  "session": { "expires_at": 1782460460000, "expires_in": 365 }
}
```

**Error:**

```json
{
  "success": false,
  "message": "Invalid credentials",
  "statusCode": 401
}
```

### Cookies

Both tokens are set as `httpOnly; SameSite=Lax; Path=/`. The frontend never reads them —
the browser sends them automatically with every request.

| Cookie          | Max-Age    |
| --------------- | ---------- |
| `access_token`  | 15 minutes |
| `refresh_token` | 365 days   |

## Architecture Decisions

**Why two file-serving roots?**
`src/assets` holds source-controlled HTML/CSS/JS templates. `public/assets` holds binary
files (images, icons) that benefit from aggressive caching and may be processed by a build
step. Keeping them separate means you can hash public assets without touching HTML.

**Why `readFileSync` at startup for the homepage?**
The file is read once and held in memory — zero disk I/O per request. For dev hot-reload,
move the read inside the handler at the cost of one disk read per request.

**Why sessions table instead of just JWTs?**
Stateless JWTs can't be revoked. The sessions table lets us invalidate a user's session
immediately on signout, and detect replay attacks (if a stolen refresh token is used after
rotation, the old jti no longer exists).

**Why same error message for wrong email vs wrong password?**
User enumeration: if the API returns different messages for "email not found" vs "wrong
password", an attacker can silently probe which emails are registered.

## V1 → V2 Migration Guide

V1 uses an in-memory Map for sessions (zero deps, instant dev setup).
V2 uses Drizzle + Postgres (production).

The only files that change:

1. `auth.service.ts` — replace `sessionStore.set/get/delete` with Drizzle `db.insert/select/update`
2. `user.repository.ts` — replace Map operations with Drizzle queries

All function **signatures, return types, and error messages stay identical**.
The controller, middleware, and frontend never change.
